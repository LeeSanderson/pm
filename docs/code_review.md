# Code Review

Reviewed on 2026-04-28. All 10 plan parts complete. Tests pass (39 backend, 13 frontend unit, 8 e2e).

Findings are grouped by priority. File paths and line numbers are given for each item.

## Medium priority

### 1. Silent details override on card update

**File:** `backend/app/board_store.py:175` and `:599–601`

`_normalize_details` replaces any empty string with `"No details yet."`. This is applied in both `add_card` and `update_card`. If a user explicitly clears a card's details field via `PATCH /api/board/cards/{cardId}`, their intent is silently overridden and the literal placeholder text is stored. The fix is to apply the default only in `add_card`, and in `update_card` allow an empty string to be stored (or stored as `""`).

### 2. No startup warning when using the default session secret

**File:** `backend/app/main.py:40,80`

```python
DEFAULT_SESSION_SECRET = "pm-mvp-dev-session-secret"

def resolve_session_secret() -> str:
  return os.environ.get("SESSION_SECRET", DEFAULT_SESSION_SECRET)
```

The default is a public string. Any deployment that omits `SESSION_SECRET` accepts session cookies signed with a known key, which means any client can forge a valid session. There is no log warning or startup assertion when the default is used. At minimum, a `warnings.warn` or `print` at startup would surface this misconfiguration. Ideally, generate a random secret at startup when the env var is absent, accepting that sessions won't survive restarts.

---

### 3. `ConversationStore` never evicts idle sessions

**File:** `backend/app/ai_board.py:117–136`

`_messages` is a plain `dict[str, list[...]]`. Sessions are added on first chat and removed only on explicit logout or if `clear()` is called. In a long-running process, sessions that never log out (e.g. browser tab closed without logout, integration tests that don't clean up) accumulate indefinitely. For the current single-user MVP this is harmless, but a session count cap (e.g. `max_sessions=1000`) or TTL would prevent the process from leaking memory over time.

---

### 4. No input length limits on user-supplied strings

**Files:** `backend/app/main.py:49–66`, `backend/app/ai_board.py:31–37`

`RenameColumnRequest.title`, `CreateCardRequest.title`/`details`, `UpdateCardRequest.title`/`details`, and `AIChatRequest.message` have no maximum length. An oversized chat message is sent verbatim to OpenRouter, which could exhaust context window or inflate costs. Pydantic supports `max_length` on `str` fields: e.g. `title: Annotated[str, Field(max_length=255)]` and `message: Annotated[str, Field(max_length=4000)]`.

## Low priority

### 5. Unreachable `json.JSONDecodeError` branch in `parse_ai_chat_response`

**File:** `backend/app/ai_board.py:177`

```python
except json.JSONDecodeError as error:
    raise AIResponseError("AI response was not valid JSON") from error
```

`AIChatModelResponse.model_validate_json` (Pydantic v2) wraps JSON parse errors inside a `ValidationError`, so a bare `json.JSONDecodeError` will never surface from that call. The catch is dead. It can be removed without changing behaviour; the `ValidationError` branch above it already handles malformed JSON.

### 6. Dead export: `createId` in `kanban.ts`

**File:** `frontend/src/lib/kanban.ts:164–168`

`createId` is exported but never imported anywhere. Card IDs are generated by the backend (`secrets.token_hex(8)` in `board_store.py`). The function can be deleted.

---

### 7. `initialData` in `kanban.ts` is never read at runtime

**File:** `frontend/src/lib/kanban.ts:18–72`

`initialData` was the in-memory seed used before backend persistence was added in Part 7. It is no longer referenced in any component or API call (only in test mocks). It can be deleted from the production module and moved inline to the test files that need it, or kept only as a test fixture.

---

### 8. `AIChatSidebar` duplicates a subset of the `AIChatOperation` type

**Files:** `frontend/src/components/AIChatSidebar.tsx:5–8`, `frontend/src/lib/boardApi.ts:3–6`

`AIChatSidebar.tsx` defines its own `AIChatSubmitResult` with `appliedOperations: Array<{ type: string }>`. The more complete `AIChatOperation` type (with the index signature) already exists in `boardApi.ts`. The sidebar only needs `.length` on this array today, but importing `AIChatResponse` from `boardApi.ts` and using it as the return type of `onSubmit` would eliminate the duplication and catch future drift.

## Test coverage gaps

### 9. No test for `update_card` with empty details

`test_app.py` covers add/edit/delete round-trips but does not verify the behaviour when `details` is sent as an empty string on a `PATCH /api/board/cards/{cardId}` call. Given finding #1 above (the silent override), this path should be explicitly tested to document the intended contract.

### 10. No test for `AIChatRequest` with an empty or whitespace-only message

`AIChatRequest` rejects blank messages via a Pydantic validator, but `test_app.py` has no test asserting that `POST /api/ai/chat` with `{"message": ""}` returns `422`. One line in `test_app.py` would cover this.

## No-action items (documented MVP constraints)

The following were flagged by review tooling but are intentional, documented trade-offs for the MVP scope:

- **Hardcoded credentials** (`user`/`password`) — documented in `AGENTS.md` and `docs/RUNNING.md`.
- **No CSRF token** — `same_site="lax"` is the standard posture for cookie-authenticated SPAs on the same origin; acceptable for a localhost MVP.
- **No rate limiting on AI routes** — single-user, local deployment.
- **No HTTPS enforcement** — localhost Docker only.
- **Chat history not persisted** — explicit constraint in `docs/PLAN.md`.
